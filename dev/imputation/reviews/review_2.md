# Review 2 — GGM Imputation Plan

## 1. Mathematical correctness
- The diagonal fix in §1.3 subtracts only one of the duplicated updates (`suf_stat_(v, v) -= delta * observations_(i, v)`), so the resulting change is just $\delta x_{\text{old}}$ instead of $x_{\text{new}}^2 - x_{\text{old}}^2$ ([dev/plans/ggm_imputation.md](dev/plans/ggm_imputation.md#L40-L52)). The text already derives $S'_{vv} = S_{vv} + \delta(2x_{\text{old}} + \delta)$, so the pseudocode should either (a) update `observations_` before the loop and write `suf_stat_(v, v) += delta * x_{i,v,\text{new}}` once, or (b) keep the current order but add $\delta^2$ after the loop (as done later in §G.1e). Without this correction, $S$ drifts whenever any diagonal entry is imputed.
- The explanation in §G.1e that `suf_stat_(variable, variable) += delta * delta` “corrects” the diagonal is correct only because `observations_(person, variable)` still stores $x_{\text{old}}$ during the loop ([dev/plans/ggm_imputation.md](dev/plans/ggm_imputation.md#L210-L245)). This dependency should be spelled out; otherwise a future refactor that updates `observations_` earlier will unintentionally double-count the diagonal. Consider encapsulating the update as `update_suf_stat_column(variable, person, delta)` so the invariants stay localized.
- $S'_{av} = S_{av} + \delta x_{ia}$ implicitly assumes $x_{ia}$ is the pre-update value for all $a$ ([dev/plans/ggm_imputation.md](dev/plans/ggm_imputation.md#L32-L48)). When multiple entries in row $i$ are imputed within one sweep, ordering matters because later conditionals depend on already-updated cells. Clarify that the sweep follows the missing-index order and uses the freshest row state, mirroring the sequential Gibbs logic already used in [src/models/omrf/omrf_model.cpp](src/models/omrf/omrf_model.cpp#L1325-L1405).

## 2. Centering interaction
- Center-once (Option A) is workable because the Gibbs draw for each missing value has conditional mean $\mu_{iv} = -\omega_{vv}^{-1}\sum_{k\neq v}\omega_{vk} x_{ik}$, and $\mathbb{E}[\mu_{iv}] = 0$ when every column remains centered. However, once imputation starts, the empirical column means of `observations_` wander; the cross-product that actually feeds $S$ becomes $S = (X - m_0)'(X - m_0) = (X - m)'(X - m) + n (m - m_0)(m - m_0)'$. With $>20\%$ missingness the $n (m - m_0)(m - m_0)'$ term can be non-negligible, biasing the posterior toward dense precision structures even though the likelihood assumes zero mean ([dev/plans/ggm_imputation.md](dev/plans/ggm_imputation.md#L63-L79); [R/validate_data.R](R/validate_data.R#L63-L102)). I would keep Option A but add instrumentation: track column means after each imputation sweep and (1) warn when $|\bar{x}_v| > 5/\sqrt{n}$, and (2) optionally recentre `observations_` and adjust `suf_stat_` when the drift exceeds a tolerance. That keeps the fast path by default yet offers a safety valve when drift would matter.
- Re-centering every iteration (Option B) is indeed costly ($O(np)$ per sweep), but documenting why Option A is acceptable—and under what regimes it might fail—will keep users from assuming the centered means are exact. Mention explicitly that posterior summaries remain valid only under MCAR/MAR and moderate missingness; the stored `column_means` are merely the initial estimates from the partially observed data.

## 3. Missing design considerations
- The plan does not cover what happens when an entire column is missing. `handle_impute()` samples from the observed values of that column, so `sample(x[-mis, node], ...)` fails if every entry is NA. We need either a guard in `validate_missing_data()` or a fallback draw (e.g., $\mathcal{N}(0,1)$) so the GGM imputer can start.
- High-missingness scenarios (>30%) and MNAR patterns are marked as “out of scope” in §6, but there is no user-facing warning. Consider logging a message when `sum(is.na)` exceeds, say, 0.2 `length(x)` or when `missing_index` concentrates in one variable. Otherwise users may trust posterior edges that are driven mostly by the prior.
- Non-ignorable missingness is not addressed. Even a short note in the documentation (or in `validate_missing_data()`) saying that the imputer assumes MCAR/MAR would prevent misuse.

## 4. Architectural fit
- Threading `na_impute` from R → `sample_ggm()` → `createGGMModelFromR()` → constructor is fine, but `set_missing_data()` should assert that `observations_` was retained; otherwise calling it after taking the sufficient-statistics constructor will silently enable `impute_missing()` with an empty matrix. Add a guard (throw or `BGMS_ASSERT`) when `observations_.n_elem == 0` and `has_missing_` is `true`.
- Because the `chain_runner` decision already depends on both `config.na_impute` and `model.has_missing_data()`, it might be simpler to let the model set `has_missing_` automatically when `missing_index_` is populated instead of exposing a separate setter. Alternatively, add a convenience constructor/factory that takes an optional missing-index matrix to keep the `na_impute` flag from propagating through every frame.
- The plan does not mention how imputation interacts with multi-chain execution. If each `GGMModel` copy stores its own `observations_`, ensure the copy constructor clones the matrix (currently it does not allocate the new members). Otherwise the chains would share mutable state.

## 5. Tests
- G.4a–G.4e focus on smoke tests and posterior correlations. They never verify that the incremental `suf_stat_` matches a freshly computed cross-product. Add a deterministic unit test that (1) runs `impute_missing()` once, (2) recomputes `X'X` from `observations_`, and (3) compares it to `suf_stat_` element-wise. That would instantly catch any diagonal bookkeeping bugs discussed above.
- Add a regression test for the centering drift: simulate data with a known non-zero mean, introduce missingness, run the imputer for a few sweeps, and assert that the column means remain within a tolerance. Without this, Option A failures would go unnoticed.
- Test the failure modes: (a) requesting `na_action="impute"` while supplying only sufficient statistics should raise a clear error, (b) columns with 100% missing values, and (c) `missing_index_nullable = NULL` while `na_impute = TRUE` (should silently skip imputation).
- Consider a stress test with 30% MCAR missingness to ensure chains still mix and `suf_stat_` stays positive definite.

## 6. Direct answer on centering
Centering matters because the likelihood treats the mean as fixed at zero; any bias in the stored column means introduces an additive term $n (m - m_0)(m - m_0)'$ in $S$, which the sampler interprets as additional covariance structure. In practice, centering once (Option A) is acceptable when missingness is light because the Gibbs conditionals keep the imputed means close to zero in expectation. Still, the implementation should (a) record the actual column means of the imputed data after sampling, (b) warn when they drift, and (c) document that users with heavy missingness should either re-center periodically or run listwise deletion. That keeps the zero-mean assumption aligned with what the C++ code really sees.
