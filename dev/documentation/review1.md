## 2. Tier boundaries

### The Tier 1 / Tier 2 line is drawn correctly

The rule — "exported = roxygen, non-exported = plain `#`" — is clean,
matches brms/rstanarm precedent, and is consistently followed in the
codebase. I checked the NAMESPACE against the source; every exported
generic and method has `#'` roxygen.

### `#' @noRd` — should any internals be promoted?

The strategy explicitly bans `#' @noRd`. This is a defensible stylistic
choice, but it has a practical cost: roxygen's `@inheritParams` only
works on functions that have roxygen blocks. If you ever want to share
parameter documentation between an internal helper and an exported
function (unlikely now, but possible), you'd need a workaround.

**Verdict:** Keep the ban. The plain-`#` `@param` pseudo-tags are
already readable enough for developers, and `@noRd` blurs the visual
signal between exported and internal code. No internals currently need
promotion.

### One exception to watch

`extractor_functions.R` has a `%||%` operator (lines 1–3) that is neither
documented nor exported. This is fine — it's a polyfill — but it sits
above the roxygen block in an odd position. Consider moving it to
`zzz.R` or `compute_utils.R` so the file opens cleanly with the
roxygen `NULL` documentation block.

## 3. Practicality

### Rules that work well in practice

- The Tier 2 comment-block template (`# ----` / `# @param` / `Returns:`)
  is already in use in `validate_data.R`, `compute_utils.R`, and
  `bgm_spec.R`. It's easy to follow and easy to grep for.
- The Tier 3 "use `omrf_model.h` as gold standard" rule is concrete and
  helpful. The OMRF header genuinely is an excellent reference.

### Rules that may cause friction

| Rule | Concern | Suggestion |
|------|---------|------------|
| **"Tag order: `@title`, `@description`, `@param`, `@return`, `@details`, `@section`, `@examples`, `@seealso`, `@export`"** | `bgm.R` currently puts `@seealso` _before_ `@details` (line 22), and `@section` blocks before `@param`. This is sensible for a 400-line roxygen block where users need cross-references early. Enforcing a rigid tag order on very long blocks may reduce readability. | Relax to: "`@param` and `@return` must come before `@examples`; other tags are ordered at the author's discretion." Or simply say "preferred order" rather than "required order." |
| **"Every `@param` must use `@inheritParams bgm` for shared parameters"** | Good principle, but `bgm()` has 37 parameters. Blindly inheriting all of them into S3 methods would generate warnings for params that don't exist on the method. `@inheritParams` is selective (only inherits params that share a name), so it's safe — but the strategy should note this. | Add: "`@inheritParams bgm` only inherits documentation for parameters whose names match the current function's formals. It is safe to use even when the receiving function has fewer parameters." |
| **"Never `\dontrun{}` unless it needs credentials or network access"** | Some CRAN-bound examples may need `\dontrun{}` for examples requiring a pre-fitted object that takes > 5 seconds to create, because `\donttest{}` is still run on CRAN's extra checks. | Refine: "Prefer `\donttest{}`. Use `\dontrun{}` only when the example cannot succeed in a clean R session (e.g., requires credentials, network, or a pre-fitted object that cannot be created quickly)." |
| **C++ `///` for all private fields** | `omrf_model.h` already uses inline `///<` on every private field (lines 225–280). `ggm_model.h` has no field-level comments. Enforcing `///` on every field is easy in new code but tedious to retrofit when fields are self-explanatory (e.g., `size_t n_;`). | Allow: "Self-explanatory fields (like `size_t n_`, `size_t p_`) may omit `///` if there's a grouped comment or the class-level doc explains the notation. Non-obvious fields (scratch vectors, caches, flags) must have `///`." |

### Contradiction with existing code

The strategy says `=` for assignment (enforced by `bgms_style.R`), but
the rule is only documented in the styler, not in the strategy. This isn't
a contradiction, but the strategy's "For AI agents" section should
mention `=` assignment and the existence of `inst/styler/bgms_style.R`
so agents don't introduce `<-`.

## 6. Specific items to flag

### Factual / technical issues

1. **`extractor_functions.R` line 6: `##'` (double hash + roxygen)**
   The first roxygen line uses `##'` instead of `#'`. This is legal
   roxygen2 syntax (the extra `#` is ignored), but it's inconsistent
   with every other file. Fix when touching this file for audit item #1.

2. **`ggm_model.h` constructor: no `@param` tags**
   The class-level Doxygen block is good (lines 10–18), but neither
   constructor has `@param` documentation. The OMRF constructor has full
   `@param` for all 10 arguments. This is the single highest-impact
   C++ documentation gap.

3. **`ggm_model.h` private methods: bare signatures**
   About 15 private methods at the bottom of the file (lines 220–260)
   have no documentation at all — no `/** */`, no `//`. These include
   mathematically non-trivial methods like `constrained_diagonal`,
   `compute_inv_submatrix_i`, `cholesky_update_after_edge`. The audit
   correctly flags this at priority #8, but I'd argue it's more urgent
   than pkgdown cosmetics.

4. **`ggm_model.h` has a TODO on line 33**
   `// TODO: we need to adjust the algorithm to also sample the means!`
   This is a design decision masquerading as a comment. It should be
   tracked in an issue or in `dev/plans/future_tasks.md`, not buried
   in a constructor initializer list.

5. **`nuts_diagnostics.R` has no file banner**
   First line is `check_warmup_complete <- function(...)` with no
   `# ====` banner and no function-level `# ----` block. The audit
   correctly flags this at #12 but understates the gap — the file has
   zero structured comments.

6. **Strategy mentions `@family diagnostics` for `extract_rhat`,
   `extract_ess`** but these are currently in the `extractor_functions`
   `@rdname` group. If they get `@family diagnostics`, they'd also need
   to be pulled out of the shared Rd page or you'd have conflicting
   `@family` tags on the same page. Decide: either keep them on the
   shared page with `@family extractors`, or give diagnostics their own
   page. I recommend keeping them together with `@family extractors`
   for now; add a separate `@family diagnostics` only if the number of
   diagnostic functions grows.

### Style / consistency nits

7. **Assignment style in `extractor_functions.R`**: The file mixes `=`
   (lines 37, 42) and `<-` (line 59: `center_continuous_data <-` — wait,
   that's in `validate_data.R`). Actually, `extractor_functions.R`
   consistently uses `=` for assignment, which is correct per
   `bgms_style.R`. However, `validate_data.R` uses `<-` throughout
   (lines 59, 97, 107, etc.). This is a real inconsistency. Run
   `styler::style_file("R/validate_data.R", style = bgms_style)` as a
   cleanup step.

8. **`validate_data.R` uses both `function(x)` and `function(x) {`
   styles** — this is fine by the strategy (no rule on brace placement
   for one-liners), but worth noting for consistency.
